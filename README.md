## рҹҡЁ The Problem

Campus fundraising runs on **trust** вҖ” and trust keeps failing.

Student clubs collect money via UPI links and WhatsApp messages with zero accountability. Donors have no way to verify if their вӮ№500 ever reached the cause. Goals are never enforced вҖ” a campaign raises 30% of its target and spends it anyway. There is no refund mechanism, no audit trail, and no recourse.

| What's broken | The reality |
|---|---|
| **Fund accountability** | Money collected via UPI with no tracking after transfer |
| **Goal enforcement** | Campaigns spend partial funds with zero donor protection |
| **Transparency** | Only the treasurer knows where the money went |
| **Trust** | Every new campaign starts from zero credibility |
| **Refunds** | Non-existent вҖ” donors have no recourse if goal fails |

> *47 lakh registered NGOs in India. Less than 10% file audited accounts.*

---

## вң… Our Solution

**CampusFund fixes this at the infrastructure level, not the policy level.**

Every donation is locked in an **Algorand escrow wallet** the moment it's sent. The funds are mathematically unreachable until the campaign hits its goal вҖ” no human can touch them early, no admin can override it, no exception exists. When the goal is met, the campaign creator can claim. If it fails, donors are refunded. Automatically. By code.

```
Donor sends вӮ№500   вҶ’   Algorand locks it in 4 seconds at ~0.001 ALGO fee
Goal reached       вҶ’   Creator claims instantly, on-chain sweep
Goal fails         вҶ’   Donors refunded automatically, zero friction
Anyone audits      вҶ’   Every rupee on public ledger, forever
```

## рҹ”— Algorand Escrow Design

This is the core technical innovation of CampusFund. Each campaign deploys a custom Algorand smart contract that enforces invoice-gated fund release вҖ” no trusted intermediary required.

### How the escrow works

```
1. Campaign created
   в””в”Җв–¶ Server deploys a unique Algorand smart contract per campaign
   в””в”Җв–¶ Contract address (escrowAddress) stored on Campaign record in DB
   в””в”Җв–¶ Contract enforces all release logic on-chain вҖ” server cannot
       unilaterally sweep funds

2. Donor sends a contribution
   в””в”Җв–¶ Client: Pera Wallet signs a payment txn to escrowAddress
   в””в”Җв–¶ Donor's ALGO moves into the contract escrow (4 second finality)
   в””в”Җв–¶ Server: records Donation row in DB with Algorand txId
       (donor wallet address stored for pro-rata refund calculation)

3. Goal check (on each donation)
   в””в”Җв–¶ Server fetches live balance from Algod: escrowAddress balance
   в””в”Җв–¶ If balance >= campaign.goalALGO вҶ’ campaign.status = "funded"
   в””в”Җв–¶ Invoice submission unlocks for the creator

4. Creator submits invoice(s)
   в””в”Җв–¶ Creator uploads one or more invoices via /dashboard
   в””в”Җв–¶ Each invoice specifies an amount вүӨ remaining escrow balance
   в””в”Җв–¶ Invoices are stored and made visible to donors (transparency layer)
   в””в”Җв–¶ Smart contract validates the claimed amount matches invoice total

5. Creator claims (per invoice)
   в””в”Җв–¶ POST /api/campaigns/[id]/claim
   в””в”Җв–¶ Smart contract verifies: invoice amount вүӨ on-chain escrow balance
   в””в”Җв–¶ Contract releases exactly the invoiced ALGO вҶ’ creator's wallet
   в””в”Җв–¶ Multiple invoices can be submitted and claimed incrementally
   в””в”Җв–¶ Campaign status вҶ’ "claimed" once fully settled

6. Excess funds / Goal failure
   в””в”Җв–¶ If goal met but invoiced total < escrowed balance:
       remaining funds are refunded to donors pro-rata
       (each donor receives back: excess Г— (their contribution / total raised))
   в””в”Җв–¶ If deadline passes with balance < goal вҶ’ status "expired"
       full escrow balance refunded to donors pro-rata
   в””в”Җв–¶ Smart contract enforces both refund paths on-chain
```
